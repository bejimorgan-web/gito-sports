# PUBLISHED_STREAM_PERSISTENCE_AUDIT

## Summary

The published live feed is persisted in `apps/backend/src/db/connection.ts` via an SQLite file at `env.databasePath` and exposed through `apps/backend/src/repositories/operations-repository.ts`.

The disappearing published stream is not caused by an explicit startup cleanup or auto-delete routine. Instead, the live feed is generated by a filtered query and depends on the exact persisted state of `streams`, `matches`, and provider/channel metadata.

## Reproduction

1. Create a match
2. Assign a stream
3. Approve the stream
4. Publish live

Verify rows in:
- `matches`
- `streams`
- `scheduling_matches`
- `match_streams`

Restart backend.
Restart desktop app.

Verify again:
- Rows still exist?
- Statuses changed?
- Published feed API result?
- Mobile feed API result?

## Findings

### 1. Persistence model in code

The canonical published state lives in:
- `streams.published_at` (timestamp)
- `streams.status = 'active'`
- `matches.status = 'published'`

This model is implemented in:
- `apps/backend/src/repositories/operations-repository.ts#publishStream`
  - updates `streams` with `published_at` and sets status/approval_status to `active`
  - updates `matches.status` to `published`
- `apps/backend/src/repositories/operations-repository.ts#listPublishedLiveMatches`
  - returns only streams where
    - `s.status = 'active'`
    - `s.published_at IS NOT NULL`
    - `m.status = 'published'`
    - `s.health_status != 'failed'`
    - `c.status = 'active'`
    - `p.status = 'active'`
    - `p.availability_status != 'offline'`

The UI consumer uses:
- `/live-matches/current`
- `/mobile/matches/live`

Both endpoints call `listPublishedLiveMatches()`.

### 2. Startup routines

Only startup routine found:
- `apps/backend/src/startup/validateUploads.ts`
- Imported and run by `apps/backend/src/server.ts`

There are no backend startup cleanup services, no stream recovery service, and no health monitoring service in `apps/backend/src/startup`.

### 3. Legacy vs current data model

There are two separate match models in the backend:
- `matches` / `streams` used for publish lifecycle and live feed
- `scheduling_matches` / `match_streams` used by older scheduling/stream-assignment APIs

The published live feed does not join `scheduling_matches` or `match_streams`.
That means published state is not derived from those tables in the current live feed implementation.

### 4. Potential root causes for disappearance

Two likely causes:
1. **Different SQLite file on restart**
   - `DATABASE_PATH` defaults to `data/gito.sqlite`
   - relative path resolution is done from `process.cwd()` in `apps/backend/src/db/connection.ts`
   - if backend restarts from a different working directory, a different DB file may be opened and appear empty
2. **Feed filtering hides data**
   - published rows remain in DB but are filtered out when any of these conditions change:
     - stream status is not `active`
     - stream health becomes `failed`
     - match status is not `published`
     - channel/provider status is not `active`
     - provider availability status is `offline`

## Answers

### A. Why does published content disappear after restart?

Because the live feed is produced by a query with strict state filters rather than a simple existence check.
The backend persistence path may also point to a different SQLite file on restart if `DATABASE_PATH` is resolved relative to the current working directory.

### B. Which code path removes or hides it?

The hiding path is in `apps/backend/src/repositories/operations-repository.ts#listPublishedLiveMatches`.
It filters published content by:
- `s.status = 'active'`
- `s.published_at IS NOT NULL`
- `m.status = 'published'`
- `s.health_status != 'failed'`
- `c.status = 'active'`
- `p.status = 'active'`
- `p.availability_status != 'offline'`

There is no startup cleanup route removing it, so it is most likely hidden by filtering rather than deleted.

### C. Is data deleted or merely filtered?

Most likely it is **filtered**.
No startup cleanup service deletes published records on startup.
If the backend opens a different SQLite file after restart, then the original rows are still present in the original DB, but the restarted service is reading a different database.

### D. Correct persistence model

The correct model is:
- Persist published state in `streams.published_at`
- Keep `streams.status = 'active'` and `matches.status = 'published'`
- Expose live feed from `streams JOIN matches` only
- Treat `scheduling_matches` and `match_streams` as legacy scheduling/assignment tables, not the source of truth for published live content

## Recommended fixes

1. Resolve `DATABASE_PATH` to an absolute path in `apps/backend/src/config/env.ts` or `apps/backend/src/db/connection.ts`.
   - e.g. normalize relative paths against `apps/backend` package root or workspace root
2. Keep the published feed query focused on persisted publish state, and avoid accidental filtering by stale provider/channel metadata.
3. Add a startup validation step for the DB file path and database version, to ensure restarts reuse the same file.
4. Clarify the legacy model gap: if `scheduling_matches`/`match_streams` must represent the same published match, implement explicit synchronization or migrate to the new `matches`/`streams` model.
